# Content Centralization Plan вҖ” Move Everything to Supabase

**Goal:** Single source of truth. No more scattered content, no more manual syncs, no more deployment failures.

**Status:** Implementation Complete вҖ” Pending Migration Push & Testing

---

## Current State (Broken)

| Location | Purpose | Problem |
|---|---|---|
| `story.js` | Legacy narrative with functions | Outdated, not bundled |
| `packages/story-content/src/storyData.js` | Bundled static content | Manually updated, gets out of sync |
| `apps/admin` вҶ’ localStorage | Admin edits during authoring | Ephemeral, must export manually |
| Admin exports (JSON files) | Snapshots for import | Scattered in repo, error-prone |
| `packages/story-content` bundled into apps | Build payload | Adds no value, makes deploys fragile |
| Git history | Manual snapshots of content | Mixing content updates with code changes |

**Why it breaks:**
- Admin makes 50 edits вҶ’ saves to localStorage only
- Author forgets to export or export lands in wrong place
- Build reads stale `storyData.js`
- Player and admin diverge
- Redeploy loses changes or ships broken state

## Implementation Phases

### Phase 1: Supabase Schema (Week 1)

Create tables in Supabase:

```sql
-- Stories table: story content (days, envelopes, choices, flow rules)
CREATE TABLE stories (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW(),
  
  -- Content JSON
  prologue JSONB,         -- prologue lines + signoff
  days JSONB,             -- array of day objects
  flowMap JSONB,          -- rules for routing
  
  -- Publishing
  is_published BOOLEAN DEFAULT FALSE,
  published_at TIMESTAMP,
  
  -- Metadata
  author_id UUID,
  version INT,
  change_notes TEXT
);

-- History table: track all changes
CREATE TABLE story_versions (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  story_id BIGINT REFERENCES stories(id),
  created_at TIMESTAMP DEFAULT NOW(),
  change_notes TEXT,
  prologue JSONB,
  days JSONB,
  flowMap JSONB
);

-- Player responses (already using this table, keep as-is)
ALTER TABLE player_responses 
  ADD COLUMN story_version_id BIGINT REFERENCES story_versions(id) IF NOT EXISTS;
```

**Done:** Tables created, migrations committed, RLS policies set

### Phase 2: Move Admin to Supabase (Week 1)

**Current flow:**
```
Admin edits вҶ’ localStorage вҶ’ manual export вҶ’ JSON file вҶ’ git commit вҶ’ rebuild
```

**New flow:**
```
Admin edits вҶ’ Supabase (real-time) вҶ’ Player fetches live version
```

**Changes:**
1. Replace `adminStorage.js` with `supabaseStorage.js`
   - `loadAdminDraft()` вҶ’ fetch from `stories` table (draft version, `is_published = false`)
   - `saveAdminDraft()` вҶ’ update `stories` table
   - `createAdminSnapshot()` вҶ’ insert into `story_versions` table
   - Remove localStorage entirely (or keep as local cache only)

2. Add "Publish" button to admin
   - `publishStory()` вҶ’ set `is_published = true`, record timestamp
   - Creates immutable `story_versions` entry
   - Player receives new version immediately

3. Add version history UI
   - Show published versions with timestamps
   - Allow rollback to previous version
   - Show change notes

**Done:** Admin reads/writes to Supabase, no more localStorage persistence

---

### Phase 3: Move Player to Supabase (Week 1вҖ“2)

**Current flow:**
```
Player app imported with bundled storyData.js вҶ’ static content
```

**New flow:**
```
Player app вҶ’ fetch published story from Supabase вҶ’ real-time updates
```

**Changes:**
1. Add `fetchPublishedStory()` API call in player
   ```js
   const { data: story } = await supabase
     .from('stories')
     .select('*')
     .eq('is_published', true)
     .single();
   ```

2. Remove `packages/story-content` (no longer needed)
3. Remove all bundled content files
4. Update player to handle network failures gracefully
   - Cache last fetched content in localStorage
   - Retry on app launch

**Done:** Player fetches live content on startup, no stale bundled content

## Benefits

| Problem | Solved By |
|---|---|
| Multiple versions scattered across files | Single Supabase table |
| Manual exports/imports | Auto-saving on every edit |
| Deployments that don't include latest content | Real-time fetching from Supabase |
| Admin and player getting out of sync | Both read from same source |
| Lost changes when failing to export | Version history + audit trail |
| Build pipeline complexity | No content bundling |
| Redeploys that wipe local edits | Content is independent of Vercel |

---

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Network failure вҶ’ player can't fetch | Cache last version locally, continue with stale |
| Supabase down | Player still works (offline), uses cache |
| Admin publishes broken content | Add preview before publish, version history for rollback |
| Old player apps with bundled content | Serve current version from Supabase, graceful fallback |

## New Author Workflow

**Before (broken):**
1. Author edits story in admin вҶ’ localStorage only
2. Author manually exports JSON to `Backup Scripts/` folder
3. Author copies JSON to `packages/story-content/src/storyData.js`
4. Author commits changes to git
5. Build bundles storyData.js
6. Vercel deploys with bundled content
7. Player loads static bundled content
8. Admin and player can diverge
9. Redeploy loses local edits

**After (fixed):**
1. Author opens admin app вҶ’ loads draft from Supabase automatically
2. Author edits story вҶ’ auto-saves to Supabase every 2 seconds
3. Author clicks "Publish" button вҶ’ creates immutable version, sets `is_published = true`
4. Player app loads next time вҶ’ fetches published story from Supabase (real-time)
5. Author can click "Version History" вҶ’ see all published versions
6. Author can click "Rollback" вҶ’ restore previous version as new draft
7. Git history is kept clean (only code changes, not content dumps)
8. Vercel deploys only front-end code (no content payload)
9. Content updates are independent of deploys

**Key differences:**
- вң… No more bundled content in git
- вң… No more manual exports/imports
- вң… No more sync failures
- вң… Instant updates (player fetches live version)
- вң… Version history built-in
- вң… Deployments are simpler (no content to bundle)
